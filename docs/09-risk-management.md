# 09. 리스크 관리

> **이 문서는 이 프로젝트에서 가장 중요한 문서다.**
> 전략이 망해도 돈은 복구할 수 있지만, 리스크 관리가 무너지면 원금이 날아간다.

## 기본 원칙

1. **먼저 잃지 않는다, 그 다음 번다.**
2. **시스템은 의심스러우면 멈춘다.** 자동 재시도·자동 복구보다 **안전 정지(fail-safe)** 우선.
3. **사전 검증(pre-trade) > 사후 수정.** 잘못된 주문은 취소보다 내지 않는 것이 낫다.
4. 모든 한도는 **설정으로 외부화**하여 코드 수정 없이 조정 가능.
5. **모의투자에서 먼저 검증**. 실전 전환 기준은 [14-roadmap.md](./14-roadmap.md).

## 포지션 사이징

### 접근법 비교

| 방식 | 설명 | 장단점 |
|---|---|---|
| 고정 비율 | 종목당 자본의 N% | 단순, 예측 가능. 변동성 무시 |
| 변동성 타깃 | 종목 변동성의 역수로 비중 | 리스크 균등. 추정 오차 민감 |
| Kelly 기준 | 승률·손익비 기반 최적 비중 | 이론상 최적. **실전에선 절반/사분위 사용 권장** |
| ATR 기반 | ATR 대비 손실 허용액으로 크기 결정 | 손절폭과 연동. 단기 전략에 유용 |

**선택 (MVP)**: 고정 비율 (종목당 5%) + 전략별 합산 한도

**확장**: 변동성 타깃팅 도입 (과거 20일 수익률 표준편차 기반)

## 익스포저 한도

| 한도 | 기본값 (개인 자본 기준) |
|---|---|
| 종목당 최대 비중 | 10% |
| 섹터당 최대 비중 | 30% |
| 단일 전략 최대 자본 사용 | 50% |
| 총 투자 비중 (현금 대비) | 80% (현금 버퍼 20%) |
| 신용 · 대주 | **금지** (MVP) |

## 손절 / 익절

- **손절(Stop-loss)**: 진입가 대비 -5~-8% (전략별 조정)
- **트레일링 스톱**: 최고가 대비 -10% (스윙)
- **익절**: 규칙 기반 (목표가 도달, 지표 반전, 시간 경과)
- **시간 기반 청산**: 인트라데이는 장 마감 전 전량 청산

**주의**: 손절은 **의도대로 체결되지 않을 수 있음** (갭 하락, 거래정지). 시스템이 손절가를 넘어도 기계적 전량 시장가 매도만 신뢰해서는 안 됨.

## 서킷 브레이커 (Kill Switch)

시스템이 스스로 멈춰야 하는 상황:

| 조건 | 동작 |
|---|---|
| **일일 손실 한도 초과** (예: 자본의 -3%) | 전 전략 신규 진입 중단, 보유 포지션 유지 |
| 누적 MDD -15% 도달 | 당일 매매 중단 + 알림 |
| 이상 주문 패턴 (1분 내 10건 초과) | 즉시 정지 + 수동 승인 필요 |
| Broker API 응답 실패 5회 연속 | 매매 중단, 알림 |
| 시세 데이터 5분 이상 지연 | 매매 중단 |
| 시장 이벤트 (서킷브레이커, VI 빈발) | 보수 모드 전환 |

**복구는 항상 수동**. 자동 재개는 금지.

## 사전 검증 체크리스트 (Pre-Trade Checks)

주문 생성 전 Risk Engine이 반드시 확인:

- [ ] 종목 거래 가능 상태 (거래정지·관리·투자주의 등)
- [ ] 주문가가 호가 단위에 맞음
- [ ] 주문가가 상·하한가 범위 내
- [ ] 주문 수량이 보유 수량 이하 (매도 시)
- [ ] 주문 금액이 가용 현금 이하 (매수 시)
- [ ] 종목·섹터·전체 한도 내
- [ ] 일일 손실 한도 미초과
- [ ] 동일 신호 중복 주문 아님 (멱등키 검사)
- [ ] 장 운영 시간 내

실패 사유는 모두 **로그에 남김**. 통과 사유는 기본 정보만 기록.

## 주문 실행 안전장치

- **시장가 주문 비중 최소화**: 슬리피지 큼. 유동성 낮은 종목은 금지
- **분할 주문(splitting)**: 대형 주문은 여러 건으로 쪼개 시장 충격 완화
- **주문가 이상치 감지**: 현재가 대비 ±N% 초과 시 차단
- **1분 내 최대 주문 수 제한**

## 감사 로그

모든 리스크 이벤트는 **append-only** 로그에 기록:
- 신호 → 사전 검증 결과 → 주문 생성/차단 → 체결/실패
- 차단 사유는 **코드(enum)**와 **설명**을 함께

로그는 PnL 재현·사후 분석·시스템 개선의 근거가 된다.

## 시나리오별 대응

### 1. 포지션이 이상하게 벌어졌다
- 원인: 부분 체결 미반영, 장 외 체결, 잘못된 수량 주문
- 대응: 장 시작 전 **증권사 잔고와 내부 상태 정합성 검증** 루틴. 불일치 시 매매 중단 + 알림

### 2. 시스템 장애 중 포지션 보유
- 대응: **보유 포지션은 유지**, 신규 진입 중단. 수동으로 청산 판단

### 3. Flash Crash / 이상 호가
- 대응: 급격한 가격 변동(1분 ±3% 초과) 감지 시 해당 종목 매매 중단

### 4. 계좌 해킹 의심
- 대응: 내부 시스템에서 주문하지 않은 체결 감지 시 **즉시 API 키 폐기** + 비밀번호 변경

## 개인 심리 관리

- 시스템이 손실 중이라도 **수동 개입 금지**를 원칙으로 함
- 개입하려면 **기록**을 남길 것 (언제·왜·결과). 반복 개입이 보이면 시스템 규칙에 반영
- 일일 손익을 **실시간으로 보는 습관 지양** — 일 단위 요약으로 충분

## 테스트

- 리스크 엔진은 **단위 테스트 커버리지 최우선** (전략 테스트보다 앞)
- 엣지 케이스: 상한가·하한가·거래정지·소수점 수량·가용현금 경계
- 모의투자에서 의도적으로 한도 초과 주문 발생시켜 **차단 동작 검증**
