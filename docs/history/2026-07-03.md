# 2026-07-03

**요약**: Docker 재시작 과정에서 컨테이너·이미지·볼륨이 전부 초기화되어 PostgreSQL
`pgdata` 볼륨(15만 행)이 소실. `data/kronos.db`에서 재마이그레이션으로 복구하고,
재발 방지를 위해 PG 데이터를 **호스트 디스크 bind mount**로 전환.

## 한 일

### PG 볼륨 소실 → 복구
- 사용자가 Docker 재시작 후 컨테이너 재기동 요청. 점검 결과 **볼륨 목록이 완전히 비어
  있고**(`pgdata` 소실) 컨테이너·이미지도 전무 → Docker 데이터 초기화 정황 확인.
  (동시에 레지스트리 네트워크 이슈는 회복됨 — `docker pull hello-world` 성공)
- 원본 `data/kronos.db`(07-02 01:11 복구 스냅샷)가 호스트에 생존해 있어 복구 가능 판단.
- 빈 postgres 기동 후 `scripts/migrate_sqlite_to_pg.py`로 재적재 → 전 테이블 건수 일치
  (news 126,337 / disclosures 25,550 / tickers 3,965 / collector_runs 149,574 /
  ticker_aliases 7 / sentiments 11,811).
- 호스트 워커 4종(collector/sentiment/classify/dashboard) 재기동. Ollama는 brew
  네이티브라 무관하게 생존(HTTP 200).

### 재발 방지: PG 데이터를 호스트 디스크로
- `docker-compose.yml`: postgres 볼륨을 named volume `pgdata` →
  **bind mount `./data/pgdata`**로 변경. Docker 초기화(reset/purge)에도 데이터 생존.
- `data/`는 이미 gitignore됨.

## 결정

- **bind mount 채택 이유**: named volume은 Docker 초기화 시 함께 삭제되어 이번 사고의
  직접 원인. 호스트 디스크 bind mount는 Docker 상태와 독립적으로 생존.
- **SQLite 손상 재발 우려 없음**: 과거 손상은 collector+sentiment+dashboard 3개가 같은
  SQLite를 bind mount로 동시 접근(VirtioFS WAL 락 미보장)한 것이 원인. PostgreSQL은
  단일 postmaster 프로세스가 데이터 파일을 독점하고 클라이언트는 소켓/TCP로 접속 →
  다중프로세스-네트워크FS 손상 모드가 원천적으로 없음. initdb·기동 정상 확인.
- **런타임은 하이브리드 유지(B안)**: 복구·데이터 안정화 우선. 레지스트리가 회복되어
  완전 컨테이너화도 가능하나, classify 서비스를 compose에 추가한 뒤 별도로 전환하기로.
